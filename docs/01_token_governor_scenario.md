# Сценарий Hermes Token Governor

## Цель

Hermes должен сначала определять сложность задачи, а только потом выбирать модель, память, инструменты и агентов.

Схема:

Пользователь -> Task Router -> Budget Guard -> Memory Selector -> Model Selector -> Execution -> Reviewer -> Answer -> Checkpoint.

## Уровни задач

### L0 — без модели

Команды, статусы, запись задач, простые выборки из базы. Выполняются обычным кодом.

### L1 — простая задача

Один дешёвый вызов модели. Без агентов и без памяти.

Примеры: перевод, короткое письмо, простое извлечение полей, переформулировка.

### L2 — средняя задача

Один исполнитель, иногда проверка результата. Память ограничена несколькими релевантными фрагментами.

Примеры: разобрать заявку клиента, структурировать письмо, сравнить позиции с базой, подготовить краткое ТЗ.

### L3 — сложная задача

Planner + worker + reviewer. Не более трёх агентов и не более двух раундов.

Примеры: архитектура, микросервисы, пайплайн, классификация тендера, задание для Codex.

### L4 — проектная задача

Для многофайлового анализа, кода, CI/CD, RAG и длительных процессов. Обязательны checkpoint summaries.

## Политика памяти

Память делится на слои:

1. Hot memory — краткий профиль и активный проект.
2. Project summary — сжатая память проекта.
3. Skill memory — инструкции по типу задачи.
4. Archive memory — только через поиск.
5. Raw files — только при прямой необходимости.

Правило: для L1 память по умолчанию выключена.

## Политика агентов

Агенты включаются только если есть хотя бы одно условие:

- три и более независимых подзадачи;
- несколько файлов;
- нужен спор вариантов;
- нужен reviewer;
- пишется код;
- цена ошибки выше цены токенов.

Агенты не нужны для перевода, короткого письма, простого расчёта и обычной выжимки.

## Checkpoint

Если сложная задача достигла лимита, Hermes создаёт краткое состояние:

- что сделано;
- какие решения приняты;
- какие вопросы открыты;
- что делать следующим шагом.

Следующий запуск должен использовать checkpoint, а не всю историю.

## Порядок внедрения

1. Входящее сообщение сначала идёт в Task Router.
2. Router возвращает уровень L0-L4.
3. Budget Guard назначает лимиты.
4. Memory Selector выбирает нужные skills и фрагменты памяти.
5. Model Selector выбирает класс модели.
6. Execution запускает обычный ответ или multi-agent flow.
7. Usage Logger пишет фактический расход.
8. Для сложных задач сохраняется checkpoint.
