# Сценарий второго бота: Code Guard Bot

## Цель

Второй бот должен следить за написанием кода, проверять изменения, спорить с исполнителем до трёх раундов и обращаться к человеку, если согласия нет.

Рабочее имя: `Code Guard Bot`.

## Роли

### 1. Implementer

Пишет код по задаче. Может быть Codex, Hermes worker или другой coding-agent.

### 2. Code Guard

Проверяет diff, тесты, архитектуру, безопасность, соответствие ТЗ и экономию токенов.

### 3. Supervisor

Собирает результат спора. Если спор не решён за три раунда — формирует вопрос пользователю.

## Основной сценарий

1. Пользователь ставит задачу.
2. Hermes Router классифицирует её как L3 или L4 coding task.
3. Implementer делает изменения в отдельной ветке.
4. Создаётся Pull Request.
5. Code Guard получает diff и чек-лист.
6. Code Guard даёт один из статусов:
   - `APPROVE`
   - `REQUEST_CHANGES`
   - `NEED_HUMAN_DECISION`
7. Если есть замечания, Implementer отвечает или исправляет.
8. Спор повторяется максимум 3 раунда.
9. Если согласия нет, Supervisor отправляет пользователю краткий выбор.

## Лимит спора

Раунд считается завершённым, когда есть пара:

- позиция Implementer;
- позиция Code Guard.

Максимум: 3 раунда.

После третьего раунда нельзя продолжать спор. Нужно спросить человека.

## Формат вопроса пользователю

```text
Нужна ваша позиция по коду.

Задача: ...

Implementer предлагает: ...

Code Guard возражает: ...

Риск, если принять Implementer: ...

Риск, если принять Code Guard: ...

Варианты:
1. Принять код как есть.
2. Исправить по Code Guard.
3. Остановить задачу и переписать ТЗ.
```

## Чек-лист Code Guard

Code Guard проверяет:

- код решает исходную задачу;
- нет прямого пуша в main;
- изменения минимальны;
- нет лишней сложности;
- не сломаны существующие сценарии;
- нет утечек ключей и секретов;
- есть тест или понятная причина, почему тест невозможен;
- есть логика ошибок;
- соблюдены лимиты токенов;
- новые промты не заставляют модель грузить всю память;
- multi-agent включается только для L3/L4.

## Правило остановки

Если после 3 раундов нет согласия:

- спор прекращается;
- создаётся короткий отчёт;
- пользователю отправляется вопрос;
- до ответа пользователя PR не мержится.

## Минимальная архитектура

```text
src/code_guard/pr_watcher.py
src/code_guard/diff_reader.py
src/code_guard/review_agent.py
src/code_guard/debate_manager.py
src/code_guard/human_escalation.py
src/code_guard/telegram_notifier.py
```

## Состояния

```yaml
states:
  waiting_for_pr: "Ожидаем PR"
  reviewing: "Code Guard проверяет изменения"
  changes_requested: "Нужны правки"
  debating: "Идёт спор Implementer vs Code Guard"
  approved: "Можно мержить"
  human_needed: "Нужно решение пользователя"
  blocked: "PR заблокирован"
```

## Важное правило

Code Guard не должен быть ещё одним бесконечным агентом. Он должен экономить токены:

- проверять только diff, а не весь репозиторий;
- читать связанные файлы только при необходимости;
- использовать чек-лист;
- завершать спор после 3 раундов;
- формировать короткую выжимку для человека.
