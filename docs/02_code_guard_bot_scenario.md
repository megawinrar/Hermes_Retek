# Сценарий второго бота: Code Guard Bot

## Цель

Второй бот должен следить за написанием кода, проверять изменения, спорить с исполнителем до трёх раундов и обращаться к человеку, если согласия нет.

Рабочее имя: `Code Guard Bot`.

Подробные анти-имитационные сценарии вынесены в `docs/03_code_guard_anti_imitation_scenarios.md`.

## Роли

### 1. Implementer / Bot#1

Пишет код по задаче. Может быть Codex, Hermes worker или другой coding-agent.

### 2. Code Guard / Bot#2

Проверяет diff, тесты, архитектуру, безопасность, соответствие ТЗ, экономию токенов и признаки формального выполнения.

### 3. Supervisor

Собирает результат спора. Если спор не решён за три раунда — формирует вопрос пользователю.

## Основной сценарий

1. Пользователь ставит задачу.
2. Hermes Router классифицирует её как L3 или L4 coding task.
3. Bot#1 делает изменения в отдельной ветке.
4. Создаётся Pull Request.
5. Bot#2 получает diff, список изменённых файлов, исходное ТЗ и чек-лист.
6. Bot#2 даёт один из статусов:
   - `APPROVE`
   - `REQUEST_CHANGES`
   - `FAKE_IMPLEMENTATION_DETECTED`
   - `TEST_THEATER_DETECTED`
   - `NEED_HUMAN_DECISION`
7. Если есть замечания, Bot#1 отвечает или исправляет.
8. Спор повторяется максимум 3 раунда.
9. Если согласия нет, Supervisor отправляет пользователю краткий выбор через Hermes Telegram.

## Лимит спора

Раунд считается завершённым, когда есть пара:

- позиция Bot#1;
- позиция Bot#2.

Максимум: 3 раунда.

После третьего раунда нельзя продолжать спор. Нужно спросить человека.

## Анти-имитационный контроль

Bot#2 обязан отклонить работу, если Bot#1 имитирует выполнение.

### Код не написан

Признаки:

- изменён только текст, хотя требовался код;
- нет изменений в ожидаемых исходных файлах;
- добавлена заглушка;
- логика оставлена как TODO;
- функция возвращает фиксированный ответ;
- diff не связан с исходным ТЗ.

Статус: `FAKE_IMPLEMENTATION_DETECTED`.

### Тест ради теста

Формальный тест не засчитывается.

Плохой тест:

- проверяет только импорт;
- проверяет только существование функции;
- не проверяет результат;
- полностью повторяет реализацию;
- мокает всю важную логику;
- не связан с исходным ТЗ;
- не мог бы упасть на старом коде.

Статус: `TEST_THEATER_DETECTED`.

### Пользователь попросил обернуть функцию тестами

Bot#1 обязан добавить тесты на поведение функции:

- основной успешный сценарий;
- минимум один пограничный случай;
- ошибочный сценарий, если он возможен;
- реальные входы и ожидаемые выходы;
- проверку бизнес-результата, а не только факт вызова.

Если тест не проверяет смысл функции, Bot#2 обязан отклонить работу.

## Формат вопроса пользователю

Сообщение идёт от лица Hermes, но с явной пометкой:

```text
Сообщение от Bot#2

Кратко: Bot#1 утверждает, что задача выполнена, но Bot#2 нашёл проблему.

На чём пойман Bot#1:
- ...
- ...

Риск: ...

Рекомендация Bot#2: ...

Да — согласен с Bot#2, вернуть Bot#1 на исправление.
Нет — принять работу Bot#1 как есть.
```

## Смысл кнопок

`Да` означает: согласен с Bot#2, работу не принимать, вернуть Bot#1 на исправление.

`Нет` означает: отклонить возражение Bot#2 и принять работу Bot#1 как есть.

## Чек-лист Code Guard

Code Guard проверяет:

- код решает исходную задачу;
- нет прямого пуша в main;
- изменения минимальны;
- нет лишней сложности;
- не сломаны существующие сценарии;
- нет чувствительных данных в коде и логах;
- есть тест или понятная причина, почему тест невозможен;
- тест проверяет поведение, а не просто факт наличия функции;
- есть логика ошибок;
- соблюдены лимиты токенов;
- новые промты не заставляют модель грузить всю память;
- multi-agent включается только для L3/L4.

## Правило остановки

Если после 3 раундов нет согласия:

- спор прекращается;
- создаётся короткий отчёт;
- пользователю отправляется вопрос;
- до ответа пользователя PR не мержится.

## Минимальная архитектура

```text
src/code_guard/pr_watcher.py
src/code_guard/diff_reader.py
src/code_guard/review_agent.py
src/code_guard/debate_manager.py
src/code_guard/human_escalation.py
src/code_guard/telegram_notifier.py
```

## Состояния

```yaml
states:
  waiting_for_pr: "Ожидаем PR"
  reviewing: "Code Guard проверяет изменения"
  fake_implementation_detected: "Bot#1 имитировал выполнение"
  test_theater_detected: "Тесты формальные и не проверяют поведение"
  changes_requested: "Нужны правки"
  debating: "Идёт спор Bot#1 vs Bot#2"
  approved: "Можно мержить"
  human_needed: "Нужно решение пользователя"
  blocked: "PR заблокирован"
```

## Важное правило

Code Guard не должен быть ещё одним бесконечным агентом. Он должен экономить токены:

- проверять только diff, а не весь репозиторий;
- читать связанные файлы только при необходимости;
- использовать чек-лист;
- завершать спор после 3 раундов;
- формировать короткую выжимку для человека.
