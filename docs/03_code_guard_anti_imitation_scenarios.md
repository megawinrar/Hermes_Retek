# Code Guard Bot — анти-имитационные сценарии

## Цель

Bot#2 проверяет не обещания Bot#1, а фактические доказательства: diff, изменённые файлы, тесты, результат запуска и соответствие исходному ТЗ.

Главное правило: задача не считается выполненной только потому, что Bot#1 написал `готово` или добавил любой тест.

## Сценарий A: код фактически не написан

Bot#2 должен отклонить работу, если:

- изменены только README, комментарии или промты, хотя требовался код;
- нет изменений в ожидаемых исходных файлах;
- добавлена заглушка вместо реализации;
- логика оставлена как TODO;
- функция возвращает фиксированный ответ;
- diff не содержит реализации задачи.

Статус: `FAKE_IMPLEMENTATION_DETECTED` или `REQUEST_CHANGES`.

## Сценарий B: тест ради теста

Формальный тест не засчитывается.

Плохой тест:

- проверяет только импорт;
- проверяет только существование функции;
- не проверяет результат;
- полностью повторяет реализацию;
- мокает всю важную логику;
- не связан с исходным ТЗ;
- не мог бы упасть на старом коде.

Статус: `TEST_THEATER_DETECTED`.

## Сценарий C: пользователь попросил обернуть функцию тестами

Bot#1 обязан добавить тесты на поведение функции:

- основной успешный сценарий;
- хотя бы один пограничный случай;
- ошибочный сценарий, если он возможен;
- реальные входы и ожидаемые выходы;
- проверку бизнес-результата, а не только факт вызова.

Если тест не проверяет смысл функции, Bot#2 отклоняет работу.

## Сценарий D: тесты изменены, а код нет

Если задача была исправить поведение, а изменены только тесты, Bot#2 требует доказательство, что код уже работал правильно. Если доказательства нет, работа возвращается на исправление.

## Сценарий E: спор не решён

Bot#2 спорит с Bot#1 максимум 3 раунда.

После третьего раунда Bot#2 прекращает спор и отправляет сообщение пользователю через Hermes Telegram.

## Формат Telegram-сообщения

Сообщение от Bot#2

Кратко: Bot#1 утверждает, что задача выполнена, но Bot#2 нашёл проблему.

На чём пойман Bot#1:
- пункт 1;
- пункт 2;
- пункт 3.

Риск: что произойдёт, если принять работу как есть.

Рекомендация Bot#2: принять возражение и вернуть Bot#1 на исправление.

Кнопки:
- Да — согласен с Bot#2, вернуть Bot#1 на исправление.
- Нет — принять работу Bot#1 как есть.

## Что сохранять

- task_id;
- pr_id;
- номер раунда;
- позиция Bot#1;
- позиция Bot#2;
- доказательства Bot#2;
- подозрительные тесты;
- итоговый статус;
- решение пользователя.
