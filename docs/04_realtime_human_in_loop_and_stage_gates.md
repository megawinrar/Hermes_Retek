# Realtime Human-in-the-loop и stage gates

## Цель

На средних, сложных и агентных задачах Hermes должен показывать ход работы в реальном времени и останавливаться на важных контрольных точках.

Главная идея: не спрашивать человека на каждом мелком шаге, но обязательно спрашивать перед важными решениями, которые меняют архитектуру, логику, код, тесты, бюджет токенов или бизнес-результат.

## Жёсткое правило для всех code-задач

Если в задаче есть код, значит всегда нужны:

1. тесты;
2. запуск тестов;
3. проверка результата тестов;
4. оценка рефакторинга;
5. рефакторинг, если он нужен;
6. проверка Bot#2.

Нет программных задач, которые не надо тестировать.

Если изменён код и нет тестов, stage gate не может перейти в `APPROVE`.

Если изменён код и Bot#1 не оценил рефакторинг, stage gate не может перейти в `APPROVE`.

## Режимы наблюдения

### GitHub как источник истины

GitHub хранит фактические изменения: branch, commits, pull request, diff, review Bot#2, CI/test logs и итоговое решение.

GitHub нужен для доказательств и истории, но не подходит как единственное real-time окно.

### Telegram как live-debug окно

Telegram показывает события в реальном времени:

- старт задачи;
- план;
- написан кусок кода;
- Bot#2 проверил и согласен/не согласен;
- тесты прошли/не прошли;
- нужен выбор пользователя;
- задача завершена.

Рекомендуемый вариант: отдельный Telegram topic или отдельный debug-чат `Hermes Retek DevLog`.

## Что отправлять в Telegram

Не нужно отправлять весь diff. Нужно отправлять карточки событий.

Пример без спора:

```text
[Task #123] Stage Gate: code_chunk_ready
Bot#1 написал новый кусок кода.

Что изменено:
- src/router/task_router.py
- src/router/token_budget.py

Тесты:
- добавлены тесты основного сценария;
- добавлен пограничный случай;
- тесты запущены, результат: passed.

Рефакторинг:
- оценён;
- дублирования нет;
- разделение ответственности не нарушено.

Bot#2: APPROVE
Причина: логика не противоречит ТЗ, тесты проверяют поведение.

Действие: продолжаю автоматически.
```

Если есть спор:

```text
Сообщение от Bot#2

[Task #123] Нужна ваша позиция.

Bot#1 предлагает: принять текущую реализацию.
Bot#2 возражает: код изменён, но тест проверяет только импорт, а рефакторинг не оценён.

Риск: задача будет закрыта формально, но функция может не работать.

Да — согласен с Bot#2, вернуть Bot#1 на исправление.
Нет — принять работу Bot#1 как есть.
```

## Stage gates по уровням задач

### L2 — средняя задача

Спрашивать пользователя не всегда. Обязательные уведомления:

- задача начата;
- найдено важное допущение;
- написан новый код;
- добавлены или обновлены тесты;
- тесты запущены;
- рефакторинг оценён;
- Bot#2 отклонил результат;
- задача завершена.

Обязательный вопрос пользователю только если:

- Bot#1 и Bot#2 спорят;
- меняется бизнес-логика;
- меняется схема данных;
- меняется интеграция;
- результат влияет на деньги, клиентов, поставщиков, тендеры или файлы пользователя;
- Bot#2 видит имитацию работы;
- Bot#2 видит отсутствие тестов при изменении кода;
- Bot#2 требует рефакторинг, а Bot#1 не согласен.

### L3 — сложная задача

Обязательные stage gates:

1. `plan_ready` — план готов.
2. `architecture_decision` — выбрано архитектурное решение.
3. `code_chunk_ready` — написан крупный кусок кода.
4. `tests_ready` — тесты добавлены или обновлены.
5. `tests_run_ready` — тесты запущены, результат приложен.
6. `refactor_review_ready` — рефакторинг выполнен или обоснованно не требуется.
7. `bot2_review_ready` — Bot#2 проверил.
8. `pre_merge_decision` — перед merge.

На `architecture_decision`, `refactor_review_ready` и `pre_merge_decision` пользователь спрашивается всегда, если Bot#2 не согласен с Bot#1.

### L4 — проектная задача

Пользователь спрашивается на каждом крупном этапе: архитектура, крупный PR, API, БД, логика агентов, token policy, спор по рефакторингу, слабые тесты и merge.

## Логика Bot#1 и Bot#2

### Нормальный поток

1. Bot#1 написал кусок кода.
2. Bot#1 сообщает: что изменил, зачем, какие тесты добавил, какие тесты запустил, нужен ли рефакторинг.
3. Bot#2 проверяет diff, тесты, запуск тестов и рефакторинг.
4. Если Bot#2 согласен, задача продолжается автоматически.
5. В Telegram уходит короткое уведомление.

### Bot#2 не согласен, но противоречия с ТЗ нет

Если замечание Bot#2 не противоречит исходной логике, Bot#1 исправляет автоматически.

Пример:

- Bot#2: тест слабый, добавь проверку результата.
- Bot#1: добавляет нормальный assert.
- Bot#2: нужен рефакторинг, потому что новая логика дублирует старую.
- Bot#1: выносит общую функцию.
- Пользователя не спрашиваем, только пишем в DevLog.

### Bot#2 не согласен и есть противоречие

Если замечание Bot#2 меняет смысл задачи, архитектуру, бизнес-логику или может стоить денег, система спрашивает пользователя.

## Когда спрашивать пользователя всегда

- новый сервис или микросервис;
- изменение схемы БД;
- изменение API-контракта;
- изменение логики агентов;
- изменение token policy;
- отключение или ослабление тестов;
- отсутствие тестов при изменении кода;
- спор о необходимости рефакторинга;
- рост стоимости токенов;
- спор Bot#1/Bot#2 после 3 раундов;
- Bot#2 нашёл fake implementation или test theater;
- перед merge в main.

## Commit policy

Не нужно делать commit после каждого сообщения агента.

Рекомендуемая схема:

- маленькие внутренние шаги: только Telegram event + запись в БД;
- завершённый кодовый кусок: commit в feature branch только вместе с тестами;
- готовая задача: Pull Request;
- успешная проверка Bot#2: review comment;
- решение пользователя: comment в PR + запись в БД;
- merge только после stage gate `pre_merge_decision`.

## Реализация realtime

Минимальная архитектура:

```text
Bot#1 / agents
   -> Event Bus Redis Stream
   -> Stage Gate Manager
   -> Bot#2 Reviewer
   -> Telegram DevLog Notifier
   -> GitHub PR/Commit Writer
   -> Human Decision Handler
```

Рекомендуемые модули:

```text
src/stage_gates/gate_manager.py
src/stage_gates/policy.py
src/stage_gates/event_schema.py
src/stage_gates/human_decision.py
src/realtime/telegram_devlog.py
src/realtime/event_bus.py
src/realtime/github_audit_log.py
```

## Формат события

```json
{
  "task_id": "task_123",
  "level": "L3",
  "stage": "code_chunk_ready",
  "actor": "Bot#1",
  "summary": "Добавлен Task Router L0-L4",
  "changed_files": ["src/router/task_router.py"],
  "tests_added_or_updated": true,
  "tests_run_result": "passed",
  "refactor_review": "not_needed_no_duplication",
  "needs_bot2_review": true,
  "needs_user_decision": false,
  "risk_level": "medium"
}
```

## Правило по умолчанию

Для L2: показывать в realtime, спрашивать только при риске или споре.

Для L3/L4: показывать в realtime и спрашивать на ключевых gates.

Для agent workflow: каждый агент пишет короткое событие, но пользователь видит только важные сводки и вопросы.

Для любого code change: без тестов, результата запуска и оценки рефакторинга нельзя перейти к approve или merge.
