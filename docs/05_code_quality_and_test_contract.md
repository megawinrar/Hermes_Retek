# Code Quality Contract — код всегда требует тестов

## Главный принцип

Если задача содержит изменение кода, она всегда требует:

1. проверки необходимости рефакторинга;
2. покрытия тестами;
3. запуска тестов;
4. проверки Bot#2;
5. доказательства, что тесты проверяют поведение, а не просто существуют.

Нет программных задач, для которых тесты не нужны.

## Что считается изменением кода

Кодовой задачей считается любое изменение:

- исходного кода приложения;
- конфигурации, влияющей на выполнение;
- промптов, если они влияют на поведение агента;
- схем БД и миграций;
- API-контрактов;
- логики маршрутизации задач;
- логики агентов;
- логики памяти/RAG;
- логики токенов;
- CI/CD;
- интеграций Telegram, GitHub, OpenRouter, RAG, БД.

## Обязательные тесты

Для каждого изменения кода Bot#1 обязан добавить или обновить тесты.

Минимальный набор:

- основной успешный сценарий;
- минимум один пограничный случай;
- ошибочный сценарий, если функция принимает некорректные входы или работает с внешней системой;
- проверка ожидаемого результата;
- проверка, что старое неправильное поведение больше не проходит.

## Рефакторинг

Bot#1 обязан оценить, нужен ли рефакторинг.

Рефакторинг обязателен, если:

- новая логика дублирует существующую;
- функция стала слишком большой;
- смешаны разные ответственности;
- появились скрытые зависимости;
- код сложно тестировать;
- добавлены костыли ради прохождения тестов;
- новая логика ломает архитектуру проекта.

Если рефакторинг не нужен, Bot#1 обязан кратко объяснить почему.

## Что Bot#1 обязан приложить к каждому code stage

- что изменено;
- зачем изменено;
- какие файлы изменены;
- какие тесты добавлены или обновлены;
- какие сценарии покрыты;
- как запускались тесты;
- результат запуска;
- нужен ли рефакторинг;
- что Bot#1 сделал по рефакторингу или почему он не нужен.

## Что проверяет Bot#2

Bot#2 не принимает код, если:

- нет тестов;
- тесты формальные;
- тесты не связаны с ТЗ;
- тесты проверяют только импорт или наличие функции;
- тесты не проверяют результат;
- тесты не могли бы упасть на старой реализации;
- тесты отключены, ослаблены или помечены skip без решения пользователя;
- не выполнена оценка рефакторинга;
- код стал сложнее без причины;
- Bot#1 сделал костыль вместо нормальной реализации.

## Статусы отказа

- `MISSING_TESTS_FOR_CODE_CHANGE` — код изменён, тестов нет.
- `TEST_THEATER_DETECTED` — тесты формальные.
- `REFACTORING_REQUIRED` — код требует рефакторинга.
- `WEAKENED_TESTS_DETECTED` — проверки ослаблены.
- `FAKE_IMPLEMENTATION_DETECTED` — код фактически не написан.

## Merge rule

PR нельзя мержить, если изменён код и не выполнены оба условия:

1. Bot#2 подтвердил адекватность тестов.
2. Bot#2 подтвердил, что рефакторинг выполнен или не требуется.

## Telegram rule

Если Bot#1 спорит с Bot#2 по тестам или рефакторингу больше 3 раундов, Hermes отправляет пользователю сообщение:

```text
Сообщение от Bot#2

Тема: спор по качеству кода.

Bot#1 считает, что код можно принять.
Bot#2 возражает: нет достаточных тестов / нужен рефакторинг.

На чём пойман Bot#1:
- ...

Риск: ...

Да — согласен с Bot#2, вернуть Bot#1 на исправление.
Нет — принять работу Bot#1 как есть.
```
