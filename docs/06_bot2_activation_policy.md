# Bot#2 Activation Policy — когда нужен OpenAI/Codex reviewer

## Цель

Bot#2 не должен быть постоянной дорогой прослойкой на каждом сообщении Bot#1. Он нужен там, где есть риск ошибки, имитации работы, роста стоимости токенов, нарушения архитектуры или принятия плохого кода.

Базовая схема моделей:

- Bot#1 = DeepSeek, исполнитель и рабочая сила.
- Bot#2 = OpenAI/Codex, reviewer, арбитр качества и стоп-кран.

## Главный принцип

DeepSeek может писать.

OpenAI/Codex должен принимать важные изменения.

Bot#2 не нужен для каждого сообщения Bot#1, но нужен для каждого code gate.

## Bot#2 критически необходим

### 1. Любой PR с изменением кода

Если есть code diff, Bot#2 обязателен перед approve или merge.

Проверка Bot#2:

- тесты есть;
- тесты запущены;
- тесты проверяют поведение;
- рефакторинг оценён;
- нет заглушек;
- нет ослабления старых проверок.

### 2. Новая функция

Bot#2 проверяет, что есть основной сценарий, пограничный случай, ошибочный сценарий и проверка результата.

### 3. Багфикс

Bot#2 проверяет regression test: какой тест падал до фикса и проходит после фикса.

### 4. Рефакторинг

Bot#2 проверяет, что поведение не изменилось, тесты покрывают старые сценарии, а код не стал сложнее без причины.

### 5. Архитектура

Bot#2 обязателен при новом сервисе, новой очереди, новой БД, новом cache layer, новом RAG pipeline или новом agent workflow.

### 6. Логика агентов

Bot#2 обязателен при изменении router, memory selector, model selector, agent roles, stage gates, token governor и human escalation.

### 7. Prompt changes, влияющие на поведение

Промпты, которые меняют поведение агента, считаются кодом поведения. Bot#2 обязателен.

### 8. Интеграции

Bot#2 обязателен для Telegram, GitHub, OpenRouter, OpenAI, DeepSeek, Postgres, Redis, RAG и внешних API.

### 9. Secrets, auth, permissions, user data

Bot#2 обязателен, если затронуты ключи, токены, webhooks, доступы, файлы пользователя, персональные данные или логи.

### 10. CI/CD, Docker, deploy

Bot#2 обязателен, потому что ошибка может сломать сервис или отключить тесты.

### 11. Multi-agent workflow

Если включились несколько агентов, Bot#2 обязателен как reviewer/supervisor.

### 12. Деньги, клиенты, поставщики, тендеры

Bot#2 обязателен для автоматической классификации тендеров, расчёта цены, отправки КП, обработки заявок клиента и сопоставления клиент/контакт/заказ.

## Bot#2 нужен в light-режиме

Light-режим означает: короткая проверка diff, тестов, результата запуска и refactor review без длинного reasoning.

Подходит для:

- форматирования;
- удаления unused import;
- переименования переменной;
- мелкого локального fix без бизнес-логики;
- boilerplate;
- документации, если она частично влияет на поведение.

Даже в light-режиме, если изменён код, тесты и refactor review обязательны.

## Bot#2 лишний

Bot#2 не включается для:

- L0-команд: `/status`, `/tasks`, `/done`;
- простого перевода текста;
- короткого письма поставщику;
- простой справки;
- brainstorm без кода;
- пересылки статуса CI без интерпретации;
- Telegram-уведомлений о ходе задачи;
- обычной документации, которая не влияет на поведение системы.

## Правило подключения

Bot#2 включается, если выполнено хотя бы одно условие:

- есть code diff;
- есть PR;
- изменены тесты;
- изменён prompt поведения;
- изменён token/memory/agent policy;
- изменена архитектура;
- изменена интеграция;
- затронута безопасность;
- затронуты деньги, клиенты, поставщики, тендеры;
- Bot#1 и Bot#2 спорили ранее по этой задаче;
- пользователь запросил строгий контроль.

## Правило экономии

Bot#2 не должен читать весь репозиторий без необходимости.

Порядок проверки:

1. Исходное ТЗ.
2. Changed files.
3. Diff.
4. Тесты.
5. Результат запуска тестов.
6. Ближайшие связанные файлы, если нужно.
7. Только потом дополнительный контекст.

## Итоговая формула

DeepSeek пишет и исправляет.

OpenAI/Codex проверяет и принимает.

Bot#2 не нужен для каждого сообщения, но обязателен для каждого code gate и каждого рискованного решения.
