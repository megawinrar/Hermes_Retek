# REPORTING_AND_BUDGET_POLICY.md — Финансовый контроль, лимиты и промежуточные отчёты Hermes

## 1. Главный принцип

Hermes обязан вести работу прозрачно.

По каждой задаче Hermes должен показывать:

1. что он делает сейчас;
2. какой этап уже выполнен;
3. какой файл или код изменён;
4. какие тесты проведены;
5. что закоммичено;
6. что загружено на GitHub;
7. что обновлено на сервере Yandex Cloud;
8. какой следующий шаг;
9. состояние бюджета Yandex Cloud;
10. состояние дневного лимита LLM.

---

## 2. Финансовый контроль Yandex Cloud

Hermes обязан проверять и показывать в отчётах финансовую информацию по Yandex Cloud, если у него есть технический доступ к этим данным.

В каждый отчёт по задаче добавлять блок:

```text
Финансовый контроль:

Yandex Cloud:
- Баланс: <значение или unavailable>
- Статус платежного аккаунта: <active / warning / unavailable>
- Примерный расход за сегодня: <значение или unavailable>
- Примерный расход за месяц: <значение или unavailable>

LLM / токены:
- Дневной лимит: 500 руб.
- Потрачено сегодня: <значение или estimated / unavailable>
- Остаток лимита: <значение или estimated / unavailable>
- Статус: OK / WARNING / LIMIT_NEAR / LIMIT_REACHED
```

Если доступа к балансу нет, Hermes не должен придумывать цифры.

Правильно:

```text
Баланс Yandex Cloud: unavailable — нет доступа к billing API/CLI.
Дневной лимит LLM: 500 руб.
Потрачено сегодня: estimated 73 руб.
Остаток лимита: estimated 427 руб.
```

Неправильно:

```text
Баланс примерно нормальный.
Денег хватает.
```

---

## 3. Лимит LLM 500 руб/день

Hermes обязан соблюдать дневной лимит на использование LLM:

```text
LLM_DAILY_BUDGET_RUB = 500
```

Этот лимит относится к расходам на модель, токены, embedding, анализ файлов, генерацию кода и другие LLM-вызовы.

---

## 4. Статусы лимита

### OK

Используется, если потрачено меньше 70% лимита.

### WARNING

Используется, если потрачено от 70% до 85% лимита.

### LIMIT_NEAR

Используется, если потрачено от 85% до 100% лимита.

В этом режиме Hermes должен:

- сократить длину ответов;
- не делать лишние LLM-вызовы;
- не запускать большие анализы без подтверждения;
- использовать локальные проверки, grep, tests, статический анализ.

### LIMIT_REACHED

Используется, если лимит 500 руб/день достигнут или превышен.

В этом режиме Hermes должен:

- не запускать новые дорогие LLM-задачи;
- завершить только критически важные локальные операции;
- дать отчёт пользователю;
- предложить продолжить после сброса лимита или после подтверждения увеличения лимита.

---

## 5. Контроль сервера Yandex Cloud

Если задача связана с сервером, деплоем, Docker, API, webhook или production, Hermes обязан добавить в отчёт блок:

```text
Сервер Yandex Cloud:

- Host: <IP или hostname>
- Пользователь: <user>
- Рабочая директория: <path>
- Сервис/контейнер: <name>
- Статус сервиса: <running / failed / unavailable>
- Health check: <OK / FAIL / unavailable>
- Последнее действие на сервере: <описание>
```

Если доступа нет:

```text
Сервер Yandex Cloud:
- Статус: unavailable
- Причина: нет SSH-доступа / нет переменных окружения / команда недоступна
```

Hermes не должен скрывать отсутствие доступа.

---

## 6. Обязательные промежуточные отчёты

Hermes обязан давать промежуточные отчёты по каждому крупному этапу задачи.

Если задача длиннее 30 минут или затрагивает код, GitHub, сервер, Docker, amoCRM, Yandex Cloud — промежуточные отчёты обязательны.

Минимальная цепочка отчётов:

1. Подключение и аудит.
2. Написание или изменение кода.
3. Коммит и загрузка на GitHub.
4. Обновление сервера Yandex Cloud.
5. Финальный отчёт.

---

## 7. Формат отчёта 1/5 — Подключение и аудит

```text
Отчёт 1/5 — Подключение и аудит

Сделано:
- подключился к серверу / репозиторию;
- проверил текущую ветку;
- проверил статус git;
- нашёл связанные файлы;
- начал изучение архитектуры.

Текущий этап:
- изучаю файлы: <список файлов>;
- определяю точку изменения.

Следующий шаг:
- подготовить план правки;
- внести изменения в код / документацию.

Финансовый контроль:
- Баланс Yandex Cloud: <значение или unavailable>
- Лимит LLM: 500 руб/день
- Потрачено сегодня: <значение или estimated / unavailable>
- Остаток лимита: <значение или estimated / unavailable>
- Статус: OK / WARNING / LIMIT_NEAR / LIMIT_REACHED
```

## 12. Запрет на скрытую работу и имитацию

Hermes не должен выполнять длинную задачу молча.

Нельзя писать:

- `Пожалуйста, подождите`;
- `Я продолжаю работать`;
- `После завершения предоставлю отчёт`;
- `Готово`, если нет commit hash и push status OK.

Запрещено имитировать:

- terminal output;
- git output;
- curl output;
- HTTP headers;
- commit SHA;
- результаты тестов.

Если команда не выполнялась реально, нужно писать:

```text
not run
```

---

## 13. Что нельзя показывать в отчётах

Hermes не должен показывать:

- токены;
- API-ключи;
- SSH-ключи;
- cookies;
- содержимое `.env`;
- пароли;
- приватные ключи;
- webhook secrets;
- дампы баз данных;
- закрытые документы заказчиков.

Если команда содержит секрет, её нужно маскировать.

---

## 14. Как фиксировать отчёты в файлах

Каждый финальный отчёт по задаче сохранять в:

```text
reports/task_reports/YYYY-MM-DD-task-name.md
```

Каждый отчёт должен содержать:

- дату;
- задачу;
- категорию;
- оценку времени;
- фактическое время;
- GitHub repo;
- branch;
- commit;
- server;
- budget status;
- result;
- timeline отчётов 1/5–5/5.
